# 13 — CI/CD and Environments

## Purpose

Define the three Dishton environments (`local`, `preview`, `production`), the
GitHub Actions workflows that gate every merge, the Supabase migration policy,
the secrets matrix that lists every credential per environment and where it
lives, the rollback procedure, and the deploy target. Pick **Vercel** for the
SPA host and document why; record Netlify and Cloudflare Pages as rejected
alternatives. Implementation sessions read this doc to wire pipelines and to
add or rotate secrets without reverse-engineering the system.

## Environments

| Environment | Domain | Supabase project | Vercel project | Lifetime |
|---|---|---|---|---|
| `local` | `http://localhost:5173` | `supabase start` (Docker) | `vite preview` (optional) | developer machine |
| `preview` | `dishton-pr-<n>.vercel.app` | per-PR Supabase **branch** (auto-created) | per-PR Vercel preview deployment | PR open → close |
| `production` | `dishton.app` | `dishton-prod` Supabase project | `dishton` Vercel project, `production` env | until intentionally retired |

Rules:

- `local` is the only environment where developers run migrations forward and
  back freely. `preview` and `production` are forward-only.
- `preview` Supabase branches are created from the `production` schema at PR
  open and torn down at PR close. They never share data with production; seed
  data is regenerated by `supabase db reset` against the branch.
- `production` deploys only on merge to `main`. No manual deploys. No "promote
  preview to production" shortcut — the workflow rebuilds from `main`.

## Deploy target — Vercel (chosen)

**Decision: deploy the SPA to Vercel.** Reasons:

- First-class Vite preset; zero config beyond `vercel.json` for SPA fallback.
- Built-in per-PR preview URLs that integrate with the `pull_request` GitHub
  event, which `e2e.yml` consumes to run Playwright against the actual
  preview URL rather than a local build.
- Edge network and image optimisation that we will use for the recipe hero
  images served via signed Supabase Storage URLs.
- Generous free hobby tier covering all current usage; Pro upgrade path is
  per-seat with no surprise per-request pricing.

Rejected alternatives:

- **Netlify** — also good Vite support, but its preview deploy URL is
  delivered via a separate webhook flow that complicates the `e2e.yml`
  trigger; we would re-implement the Vercel preview-resolution logic.
- **Cloudflare Pages** — strong runtime story, but its build environment
  pins Node less precisely than Vercel and has historically lagged on
  Vite 6 support; given Vite is locked in
  [02-tech-stack.md](./02-tech-stack.md), we avoid the moving target.

The deploy target may be revisited only if Vercel changes its free-tier
preview policy in a way that blocks per-PR previews; document any such change
in [15-roadmap-and-flags.md](./15-roadmap-and-flags.md) before switching.

## GitHub Actions workflows

All workflows live under `/home/user/dishton/.github/workflows/`. Three files,
one job graph:

```
pull_request
   │
   ├── ci.yml         (typecheck, lint, unit + component tests, build)
   │      │
   │      └── e2e.yml (Playwright against the Vercel preview URL)
   │
push to main
   │
   └── deploy.yml     (push migrations to Supabase + deploy to Vercel)
```

### `ci.yml`

Triggers: `pull_request` to `main`, `push` to `main`.

Jobs (one job per concern, parallelised on the same Linux runner image
`ubuntu-24.04`):

1. **`setup`** — checkout, install Node 22.x, install `pnpm` via
   `corepack enable`, restore `pnpm` cache. Outputs cache key for downstream.
2. **`typecheck`** — `pnpm typecheck` (`tsc --noEmit`).
3. **`lint`** — `pnpm lint` (`biome check .`).
4. **`test-unit`** — `pnpm test:unit`.
5. **`test-components`** — `pnpm test:components`.
6. **`test-coverage`** — `pnpm test:coverage`. Enforces the thresholds from
   [12-testing-strategy.md](./12-testing-strategy.md).
7. **`test-edge`** — installs Deno via `denoland/setup-deno@v1` and runs
   `pnpm test:edge`.
8. **`test-db`** — boots the local Supabase stack via
   `supabase/setup-cli@v1` + `supabase start`, applies migrations, runs
   `pnpm test:db`. Tears down on completion.
9. **`build`** — `pnpm build` (Vite production bundle). Uploads `dist/` as
   an artifact for the deploy workflow to consume.
10. **`migration-diff`** — runs `node scripts/check-migration-diff.mjs`
    against the PR base. It enforces the migration policy deterministically
    from the git diff: existing migration files are immutable (forward-only),
    and any change to `supabase/seed.sql` or to an existing migration must be
    accompanied by a **newly added** migration file. **Limitation:** it cannot
    detect a schema change that exists only in a developer's local database and
    was never written to a file (there is nothing in the diff to catch). A live
    `supabase db diff` against the per-PR preview branch is the backstop for
    that case; this gate guarantees the file-level invariants on every PR.
11. **`ci-success`** — an aggregation job that `needs` every other job and
    fails unless all of them succeed. This is the single status check to mark
    **required** in branch protection, so adding a job to the graph makes it
    blocking without changing repo settings.

A failure in any job blocks merge (enforced via the `ci-success` aggregate).

### `e2e.yml`

Triggers: `pull_request` to `main`, and `push` to `main`.

**Approach (chosen): run against a fully local stack, not a Vercel preview
URL.** Resolving the Vercel preview URL from a `workflow_run`-triggered job is
brittle (the deployment URL is not in that event's context, and the original
job silently passed with an empty base URL). Instead the job stands up a
deterministic, self-contained environment so it can never pass vacuously:

1. Start a real local Supabase stack (`supabase start`) — Postgres, Auth,
   Storage, and Edge Functions — and apply migrations + seed via
   `supabase db reset`.
2. Write `AI_MOCK_MODE=playwright` into `supabase/functions/.env` so the
   served Edge Functions short-circuit the Anthropic call to the canned
   fixtures in `e2e/fixtures/ai-draft.json` /
   `e2e/fixtures/ai-translation.de.json` (mirrored in
   `supabase/functions/_shared/ai/mock.ts`). No `ANTHROPIC_API_KEY` is needed.
3. Read the local API URL + anon key from `supabase status -o env`, fail the
   job loudly if the anon key cannot be resolved, and build the SPA against
   them.
4. Run `pnpm test:e2e`; `playwright.config.ts` boots `pnpm preview` and points
   the suite at `http://localhost:4173` (desktop Chrome + Pixel 5 mobile
   projects).
5. Upload the Playwright report as an artifact on failure.

**Limitation:** this exercises the app against a fresh local stack rather than
the actual Vercel preview deployment, so it does not catch Vercel-specific
edge config (headers/rewrites) — those are covered by `vercel.json` review and
the production smoke after deploy. A failure here blocks merge.

### `deploy.yml`

Triggers: `push` to `main`.

Steps (sequential, fail-fast):

1. Checkout, set up Node 22.x and `pnpm`.
2. Authenticate `supabase` CLI with `SUPABASE_ACCESS_TOKEN` and link to the
   `dishton-prod` project.
3. **`supabase db push`** — applies any new forward-only migrations from
   `supabase/migrations/`. Fails the workflow if Supabase reports drift.
4. **`supabase functions deploy`** — deploys every function in
   `supabase/functions/` to the production project.
5. **`vercel deploy --prod`** via the official Vercel CLI, using the
   `dist/` artifact from `ci.yml`.
6. Post a deployment summary comment to the merged PR with the Supabase
   migration list and the Vercel deployment URL.

There is no separate staging environment. PR previews + the locked
preview-against-AI-mock pattern cover that role.

## Supabase migration policy

- **Forward-only.** Migrations live in
  `/home/user/dishton/supabase/migrations/<UTC_TIMESTAMP>_<slug>.sql`.
  Never edit a merged migration. To change a column, write a new migration.
- **Every PR that changes the schema includes the `supabase db diff`
  output as a migration file.** The `migration-diff` job in `ci.yml`
  enforces this.
- **No destructive changes without a follow-up migration.** Dropping a
  column, removing a table, or narrowing a type requires:
  1. A migration that introduces the new shape and backfills.
  2. A code change that stops reading/writing the old shape.
  3. A second migration, in a later PR, that removes the old shape.
  Three steps, three PRs, never compressed into one.
- **RLS changes ship in the same migration as the table change.** A
  migration that adds a table without RLS policies is rejected by the
  `test-db` job (asserted in
  [12-testing-strategy.md](./12-testing-strategy.md)).
- **Seed data lives outside migrations** in `supabase/seed.sql` and runs
  only via `supabase db reset` on local and preview branches.

## Secrets matrix

Every secret is listed below with its purpose, where it is set, and which
environments need it. Names are exact.

| Secret | Purpose | local | preview | production | Stored in |
|---|---|---|---|---|---|
| `SUPABASE_URL` | SPA points here for auth + data | yes | yes | yes | `.env.local` (local), Vercel env (preview, production) |
| `SUPABASE_ANON_KEY` | SPA anon JWT for RLS-bound calls | yes | yes | yes | same as above |
| `SUPABASE_SERVICE_ROLE_KEY` | Server-side admin (Edge Functions only) | yes | yes | yes | Supabase Dashboard env per project; never in Vercel |
| `SUPABASE_ACCESS_TOKEN` | CLI auth in CI | no | yes | yes | GitHub Actions secret `SUPABASE_ACCESS_TOKEN` |
| `SUPABASE_PROJECT_REF_PROD` / `SUPABASE_PROJECT_REF_PREVIEW` | Tells `supabase link` which project (one per environment) | no | yes | yes | GitHub Actions secrets `SUPABASE_PROJECT_REF_PREVIEW`, `SUPABASE_PROJECT_REF_PROD` |
| `ANTHROPIC_API_KEY` | Anthropic API auth for Edge Functions | yes | yes | yes | Supabase Dashboard function secrets per project |
| `ANTHROPIC_MODEL` | Text lane; optional override, default `claude-haiku-4-5` | no | optional | optional | Supabase Dashboard function secrets per project |
| `ANTHROPIC_MODEL_VISION` | Vision lane; optional override, default `claude-sonnet-4-6` | no | optional | optional | Supabase Dashboard function secrets per project |
| `SENTRY_DSN_FRONTEND` | SPA error reporting | no (off) | yes | yes | Vercel env |
| `SENTRY_DSN_FUNCTIONS` | Edge Function error reporting | no (off) | yes | yes | Supabase Dashboard function secrets |
| `SENTRY_AUTH_TOKEN` | Source-map upload during build | no | yes | yes | GitHub Actions secret |
| `LOG_DRAIN_TOKEN` | Log drain target (Better Stack / Logtail; see [14-observability.md](./14-observability.md)) | no | yes | yes | Supabase Dashboard log-drain config |
| `SENTRY_ORG` / `SENTRY_PROJECT` | Sentry org + project for source-map upload | no | yes | yes | GitHub Actions secret |
| `VERCEL_TOKEN` | CI deploys to Vercel | no | yes | yes | GitHub Actions secret |
| `VERCEL_ORG_ID` | Vercel org scoping | no | yes | yes | GitHub Actions secret |
| `VERCEL_PROJECT_ID` | Vercel project scoping | no | yes | yes | GitHub Actions secret |
| `IG_OEMBED_TOKEN` | If/when IG oEmbed needs auth (feature-flagged) | no | yes | yes | Supabase Dashboard function secrets |

Naming rules:

- All client-exposed env vars must start with `VITE_` (Vite convention).
  None of the secrets above are client-exposed except `VITE_SUPABASE_URL`
  and `VITE_SUPABASE_ANON_KEY` which are aliases of `SUPABASE_URL` and
  `SUPABASE_ANON_KEY` respectively, set with both names in Vercel.
- Anything ending in `_KEY`, `_TOKEN`, or `_SECRET` is never exposed to the
  client. CI greps the built `dist/` for these substrings and fails the
  build if any leak in.

## Rollback procedure

Two flavours: a code-only rollback and a schema rollback.

### Code-only rollback (no migration to undo)

1. Revert the offending merge commit on `main`:
   `git revert -m 1 <merge_sha> && git push origin main`.
2. `deploy.yml` runs automatically against the revert commit and produces
   a fresh Vercel production deployment.
3. Confirm the production URL is healthy via the on-call runbook in
   [14-observability.md](./14-observability.md).

### Schema rollback (a migration is implicated)

The migration policy is forward-only, so "rollback" means writing a new
migration that restores the previous shape:

1. Identify the bad migration (e.g.
   `20260430_120000_drop_recipes_servings.sql`).
2. Revert the application commits that depend on the new shape.
3. Author a **reverse migration** named with a fresh timestamp that
   re-adds the dropped column / re-widens the type / restores the index,
   and includes a backfill from any data that survived.
4. Open a PR with the reverse migration plus the reverts. The standard
   `ci.yml` + `e2e.yml` gates apply.
5. On merge, `deploy.yml` pushes the reverse migration to production and
   redeploys the SPA.

Never `supabase db reset` against `production`. Never edit a published
migration file to "fix" it; write a follow-up.
