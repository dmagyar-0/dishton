# Public shareable recipe page + OG images — design

Date: 2026-06-11
Status: approved for implementation (task-directed; amends doc 00 scope, see §9)

## 1. What we're building

The landing surface of Dishton's sharing loop: a household member shares a
recipe link in a chat app → the link unfurls with a branded Open Graph card
(title, photo, servings/time) → the recipient opens a polished, public,
read-only recipe page on the app domain → a CTA funnels them into signup.

Three user-visible pieces:

1. **Share control** on the recipe detail page (editors only): toggle a
   public link on/off, copy the URL.
2. **Public recipe page** at `/r/<token>` — unauthenticated, read-only,
   Editorial Pantry styling, with servings scaling and unit toggle, plus a
   "Start your own pantry" CTA.
3. **OG unfurl**: crawlers fetching `/r/<token>` receive server-rendered
   HTML with OG/Twitter meta tags and a generated 1200×630 OG image.

## 2. Goals & non-goals

**Goals**
- Opt-in, per-recipe, revocable public links keyed by unguessable tokens.
- Link unfurls correctly in WhatsApp, iMessage (via OG), Telegram, Slack,
  Discord, Facebook, X.
- Public page reuses the existing recipe-view components and display
  pipeline (scaling + unit conversion) so the landing surface demos the
  product, not a screenshot of it.
- Zero new secrets; no API keys in the browser (unchanged invariant).
- Kill switch: a runtime feature flag can disable all public reads at once.

**Non-goals (YAGNI cuts)**
- Public household/collection pages (`/h/<slug>` stays v2 per doc 15).
- Translation toggle on the public page (translation requires an
  authenticated edge-function call; source language only).
- Comments, likes, view counts, share analytics.
- Editing or forking the recipe anonymously ("save to my pantry" lands on
  signup; post-signup import-from-share is a follow-up, not in scope).
- SEO indexing (pages are unlisted; `noindex` is set deliberately so
  share links don't turn private collections into search results).

## 3. Architecture

```
Human visitor ── /r/<token> ──► Vercel SPA catch-all ──► React route (no auth guard)
                                                          │ anon supabase-js
                                                          ▼
                                              app.get_public_recipe(token)  [RPC, SECURITY DEFINER]
                                                          │
Crawler ─── /r/<token> ──► vercel.json UA rewrite ──► Edge Function public-recipe/<token>
                                                          │ (service role → same RPC)
                                                          ▼
                                              HTML with og:* meta; og:image →
                                              Edge Function public-recipe/<token>/og.png
```

- **Humans** get the SPA. A new unauthenticated route `/r/$token` calls a
  single SECURITY DEFINER RPC with the anon key and renders the recipe.
- **Crawlers** are routed by a `vercel.json` rewrite whose `has` condition
  matches known bot User-Agents (facebookexternalhit, WhatsApp, Twitterbot,
  Slackbot, Discordbot, TelegramBot, LinkedInBot, Pinterest, Skype,
  generic `bot`) to the `public-recipe` Edge Function, which returns meta
  HTML. The rewrite is inserted **before** the SPA catch-all. The
  destination hardcodes the Supabase project domain, consistent with the
  CSP in the same file.
- **OG image** is generated by the same Edge Function at the `/og.png`
  sub-path using Satori (`og_edge`, the Deno port of `@vercel/og`),
  dynamically imported so the meta path stays light.

A misclassified human hitting the Edge Function HTML still gets a working
page: the meta HTML contains a visible link and a meta-refresh to the
canonical `/r/<token>` URL.

## 4. Data model (migration `20260611120000_recipe_shares.sql`)

```sql
create table app.recipe_shares (
  recipe_id  uuid primary key references app.recipes(id) on delete cascade,
  token      text not null unique default encode(gen_random_bytes(16), 'hex'),
  created_by uuid references app.profiles(id) on delete set null,
  created_at timestamptz not null default now()
);
```

- One active share per recipe (PK = recipe_id). Row exists ⇔ link is live.
- Revoke = delete the row; the old token dies immediately (modulo CDN/cache
  TTL on the unfurl, see §8). Re-enabling generates a fresh token.
- 128-bit random token, hex-encoded (32 chars): unguessable, unenumerable.

**RLS on `app.recipe_shares`**
- SELECT: `app.is_household_member(recipe's household)` — members see share
  status; followers and anon see nothing (the token is the secret).
- INSERT/DELETE: `app.is_recipe_editor(recipe_id)`.
- No UPDATE policy (rows are immutable; toggle = delete + insert).

**Public read RPC**

`app.get_public_recipe(share_token text) returns jsonb`,
SECURITY DEFINER, `granted to anon, authenticated`. Behaviour:

- Returns `null` when: token unknown, or runtime flag
  `feature_flags.public_recipe_shares` is disabled (kill switch).
- Otherwise returns a **whitelisted** payload only:
  `{ recipe: { title, description, servings, total_time_min,
  canonical_unit_system, source_language, source_type, source_url,
  hero_image_path, tags, ingredients[], steps[] }, household_name }`.
  No ids, no profile data, no household id, no timestamps.
- Single round-trip for SPA, Edge Function, and tests — one source of truth
  for what is publicly exposed.

**Storage access for hero images** (bucket stays private)
- New SELECT branch on `storage.objects` for `anon, authenticated`:
  object name matches `recipes.hero_image_path` of a recipe that has a
  `recipe_shares` row. The SPA public page downloads the object through
  storage RLS with the anon key (blob URL); revoking the share also cuts
  image access.
- The Edge Function fetches the hero with the service role and inlines it
  into the OG image as a data URI.

**Seed** (`supabase/seed.sql`): a share row for the seeded
"Tomato Tarte Tatin" recipe with a fixed 32-char hex token literal so local
visual validation and E2E have a deterministic public URL. Also the
feature-flag row.

## 5. Edge Function `supabase/functions/public-recipe/`

`verify_jwt = false` in `supabase/config.toml` (same mechanism as
`recipe-chat-webhook`). GET only. Routes by path suffix:

- `GET /public-recipe/<token>` → `text/html`: OG meta document.
  - `og:title` (recipe title), `og:description` (built summary, see §6),
    `og:image` (absolute URL of the `/og.png` sibling), `og:url`
    (canonical `https://<app>/r/<token>`), `og:type=article`,
    `og:site_name=Dishton`, `twitter:card=summary_large_image`,
    `robots: noindex`, `<link rel="canonical">`, meta-refresh + visible
    link to the canonical URL for misrouted humans.
  - **All interpolated strings are HTML-escaped** (title/description are
    user-controlled — XSS in the meta document is the top risk).
- `GET /public-recipe/<token>/og.png` → `image/png`, 1200×630 via
  `og_edge` (dynamic import inside the handler): paper background, Fraunces
  display title, household attribution, servings + time line in the saffron
  palette, hero photo on the right half when present (text-only layout
  otherwise), Dishton wordmark. Fraunces TTF fetched once at first render
  and module-cached.
- Unknown/revoked token or flag off → 404 JSON. Non-GET → 405.
- `Cache-Control: public, max-age=300, s-maxage=3600` on both 200 paths;
  no caching on 404.
- App origin for canonical URLs comes from a new optional env
  `PUBLIC_APP_ORIGIN` (defaults to `https://dishton.vercel.app`; set as a
  function secret in prod).

Code layout: `index.ts` (routing + handlers), `meta.ts` (pure HTML
builder), `og.tsx`-equivalent element builder kept pure so tests cover the
element tree without rasterizing.

## 6. Domain helper `src/domain/share.ts`

Pure, I/O-free, shared with Edge Functions via the `_shared/domain`
symlink:

- `shareSummary(recipe)` → `"4 servings · 55 min · 9 ingredients"` style
  OG description line (units of the canonical recipe; falls back to the
  recipe description's first 160 chars when present).
- `sharePath(token)` → `/r/<token>` (single place that knows the URL shape).

Tested to the domain 90% threshold.

## 7. SPA changes

**Share control** (`src/ui/recipe/ShareDialog.tsx` + button on the detail
page next to Edit, editors only, gated by `useFeatureFlag('public_recipe_shares')`):
- Reads share status via TanStack Query on `recipe_shares` (RLS-scoped).
- Switch on → insert row (token generated by column default; re-select to
  read it). Switch off → confirm, then delete row with a clear "old link
  stops working" hint. Copy button + `navigator.share` when available.

**Public route** (`src/routes/r/$token.tsx`):
- No `beforeLoad` guard. Validates `?servings` / `?units` search params
  exactly like the household detail route.
- Loads via `get_public_recipe` RPC (anon or authed — same path).
- Renders with the existing pieces: `RecipeImage`-style hero (via a small
  `usePublicHeroImage` hook that downloads the storage object as anon and
  returns a blob URL), `Badge` tags, `IngredientsCard`, steps list markup
  shared with the detail page, `ServingsScaler`, `UnitToggle`. No language
  toggle, no edit affordances.
- Frame: slim public header (Dishton wordmark → `/`), attribution line
  "From <household_name>'s pantry", footer CTA card "Start your own
  pantry" → `/auth/signup`. `noindex` is not needed in the SPA document
  (crawlers never see it), but the route sets `document.title`.
- Unknown token / flag off → friendly "This link is no longer active"
  state with a CTA to the homepage.
- i18n: all strings added to `i18n.en.ts` + `i18n.de.ts`.

**index.html**: add static site-level OG tags (name, description, a static
`/icons/og-default.png`) so the bare domain also unfurls.

**Feature flag**: add `public_recipe_shares` (runtime) to
`src/feature-flags/registry.ts`, seed row `enabled = true` (the feature is
opt-in per recipe; the flag is a kill switch, not a rollout gate), and the
doc-15 table.

## 8. Security & edge cases

- **Token secrecy is the access model.** 128-bit random; no enumeration
  endpoint; followers/anon cannot SELECT share rows; the RPC is the only
  anon read path and returns a whitelisted projection.
- **XSS**: meta HTML escapes every interpolated value; the SPA renders via
  React (auto-escaped).
- **Kill switch**: flag off → RPC returns null → SPA shows inactive state,
  Edge Function 404s. Storage branch stays keyed on share rows only — flag
  off doesn't hide images already shared, which is acceptable for a
  temporary kill switch; full revocation = delete rows.
- **Cache staleness**: a revoked link may unfurl from CDN cache for up to
  `s-maxage` (1 h) and PWA-cached images for up to the SW TTL; the page
  itself (RPC, NetworkFirst) dies immediately. Documented behaviour.
- **Deleted recipes/households** cascade-delete the share row.
- **Private data**: `created_by`, household id, member names never leave
  the database. `source_url` is shown as attribution (it was already
  public content).
- **CSP**: no changes needed — `connect-src` already allows the Supabase
  domain (RPC + storage download), `img-src` allows blob/https.
- **Bot rewrite misses** (unknown crawler UA) degrade gracefully: the
  crawler gets the SPA shell with the static index.html OG tags — a
  generic unfurl instead of a rich one, never a broken page.

## 10. Testing

- **DB tests** (`pnpm test:db`): member/editor/follower/anon RLS matrix on
  `recipe_shares`; RPC returns whitelisted shape for a valid token; null
  for unknown token; null when flag disabled; storage branch grants/denies
  hero reads with/without a share row.
- **Edge tests** (`pnpm test:edge`): meta HTML builder escapes
  `<script>`-laden titles; 404 unknown token; 405 non-GET; cache headers;
  OG element builder picks photo vs text-only layout. Rasterization itself
  is not unit-tested (wasm fetch); covered by manual/visual check.
- **Domain tests**: `shareSummary` permutations (no time, 1 serving,
  description fallback, truncation).
- **Component tests**: ShareDialog states (off → on → copy → revoke);
  public page renders ingredients/steps/CTA from RPC fixture; inactive
  state.
- **Visual validation** (`validating-features-visually` skill): seeded
  share token → drive signup-less `/r/<token>` at desktop + mobile,
  the share dialog flow as alice, and the adjacent recipe detail page.
- **E2E**: extend smoke with share-toggle → open public URL logged-out.
