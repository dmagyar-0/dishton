# 05 — Auth and Households

## Purpose

Specify every authenticated flow Dishton ships day-one: email/password signup
and login, session restore, password reset, the feature-flagged Google OAuth
button, the first-run "create or join household" gate, household management
(members, roles, invites), follow/unfollow other households, and the route
guards that protect everything. The implementation here uses the Supabase JS
client directly from a SPA (no SSR helper).

## Supabase client

`src/lib/supabase.ts` is the single SPA client:

```ts
import { createClient } from '@supabase/supabase-js';
import type { Database } from './database.types';

export const supabase = createClient<Database>(
  import.meta.env.VITE_SUPABASE_URL,
  import.meta.env.VITE_SUPABASE_ANON_KEY,
  {
    auth: {
      autoRefreshToken: true,
      persistSession: true,
      detectSessionInUrl: true,
    },
  },
);
```

`Database` types are generated by `supabase gen types typescript --local` and
written to `src/lib/database.types.ts`.

## Auth state

A small Zustand store mirrors the Supabase session and exposes derived flags:

```ts
// src/lib/auth.ts
import { create } from 'zustand';
import type { Session, User } from '@supabase/supabase-js';

type Profile = { id: string; display_name: string; locale: string;
  preferred_unit_system: 'metric'|'imperial'; preferred_language: string };

type Membership = { household_id: string; role: 'owner'|'editor' };

type AuthState = {
  session: Session | null;
  user: User | null;
  profile: Profile | null;
  memberships: Membership[];        // [] until first household joined
  hydrated: boolean;
  setSession: (s: Session | null) => void;
  setProfile: (p: Profile | null) => void;
  setMemberships: (m: Membership[]) => void;
};

export const useAuth = create<AuthState>((set) => ({
  session: null, user: null, profile: null, memberships: [], hydrated: false,
  setSession: (session) => set({ session, user: session?.user ?? null }),
  setProfile: (profile) => set({ profile }),
  setMemberships: (memberships) => set({ memberships, hydrated: true }),
}));
```

Bootstrap order on app start (`src/main.tsx` → `AppShell.tsx`):

1. `supabase.auth.getSession()` → `setSession`.
2. If session: `select * from app.profiles where id = auth.uid()` → `setProfile`.
3. If session: `select household_id, role from app.household_members where profile_id = auth.uid()` → `setMemberships`.
4. Subscribe to `supabase.auth.onAuthStateChange(...)` and re-run 2 + 3 on
   `SIGNED_IN`, clear on `SIGNED_OUT`.

## Route guards

TanStack Router `beforeLoad` provides typed guards. Two guard helpers in
`src/routes/_guards.ts`:

```ts
import { redirect } from '@tanstack/react-router';
import { useAuth } from '@/lib/auth';

export const requireAuth = () => {
  const s = useAuth.getState();
  if (!s.hydrated) throw new Error('auth not hydrated');
  if (!s.session) throw redirect({ to: '/auth/login' });
};

export const requireHousehold = () => {
  requireAuth();
  const s = useAuth.getState();
  if (s.memberships.length === 0) {
    throw redirect({ to: '/onboarding' });
  }
};
```

## Flows

### Signup

1. User submits `/auth/signup` (Zod-validated form).
2. `supabase.auth.signUp({ email, password, options: { emailRedirectTo: `${location.origin}/auth/callback` } })`.
   Passing `emailRedirectTo` is required so the confirmation email links back to
   the origin the user signed up from (production domain on Vercel, preview URL
   on Vercel previews, localhost in dev) instead of the project Site URL.
3. On success: profile row inserted via a database trigger
   `app.handle_new_user` (defined in [04-data-model.md](./04-data-model.md) — add it to a follow-up migration `*_profile_trigger.sql`):

```sql
create or replace function app.handle_new_user() returns trigger
language plpgsql security definer set search_path = app, public as $$
begin
  insert into app.profiles (id, display_name) values (new.id, split_part(new.email,'@',1));
  return new;
end;
$$;
create trigger on_auth_user_created after insert on auth.users
  for each row execute function app.handle_new_user();
```

4. UI redirects to `/onboarding`.
5. Email confirmation banner appears at the top of the app shell until the user
   confirms; in dev the banner is hidden because confirmations are off locally.

### Login

1. User submits `/auth/login`.
2. `supabase.auth.signInWithPassword({ email, password })`.
3. On success: `useAuth.setSession`, then router navigates to `/`.
4. Errors mapped to user-facing copy in `src/lib/auth-errors.ts`.

### Google OAuth (feature-flagged)

```tsx
{import.meta.env.VITE_FEATURE_GOOGLE_AUTH === 'true' && (
  <Button variant="ghost" onClick={() =>
    supabase.auth.signInWithOAuth({
      provider: 'google',
      options: { redirectTo: `${location.origin}/auth/callback` },
    })}>
    Continue with Google
  </Button>
)}
```

`/auth/callback.tsx` reads the session from the URL via
`detectSessionInUrl: true` and redirects to `/`.

### Password reset

1. `/auth/reset` calls `supabase.auth.resetPasswordForEmail(email, { redirectTo })`.
2. Email link returns to `/auth/callback?type=recovery` then to a "set new
   password" form.

### First-run gate

`/onboarding`:

- Two cards side by side: **Create a household** and **Redeem an invite**.
- Create flow: form `name` (required) → `insert into app.households` →
  `insert into app.household_members(role='owner')` → `setMemberships` →
  navigate to `/h/<id>`.
- Redeem flow: form `code` → `supabase.rpc('redeem_invite', { p_code: code })`
  → `setMemberships` → navigate to `/h/<id>`.

### Household members and invites

`/household/members` — table of members with role, joined date, and (for
owners) a "remove" action. The current user can leave the household; if the
last owner leaves, prompt for ownership transfer first.

`/household/invites` — list of outstanding invites (`redeemed_at is null and
expires_at > now()`), each with a one-click copy of the share URL
`https://<host>/onboarding?code=<CODE>`. "Generate new invite" calls
`supabase.rpc('create_invite', { p_household: currentId })`.

Invite codes are 8-character base32 (`A-Z2-7`), single-use, 7-day expiry — see
[04-data-model.md](./04-data-model.md). The redeem form auto-fills `code` from
the URL parameter.

### Follow / unfollow

`/household/follows`:

- Section "Following" — list of followed households with their name and member
  count. "Unfollow" deletes the row.
- Section "Add a follow" — input for a `follow code`. A follow code is
  `f_<base32-12>` and is generated via `app.create_follow_code` RPC, similar to
  invites but consumed via `app.add_follow(code)` which inserts a row into
  `app.follows`. Add this RPC alongside `redeem_invite` in the
  `*_invites.sql` migration.
- Followed households' recipes appear in `/following`, never in
  `/h/:householdId`.

### Logout

A button in the profile menu calls `supabase.auth.signOut()`, the auth store
clears, the router navigates to `/auth/login`.

## Profile screen

`/profile`:

- Editable: `display_name`, `locale` (BCP-47 picker), `preferred_unit_system`,
  `preferred_language`.
- Avatar upload writes to `recipe-images/<uid>/avatar.jpg` (reuses the
  bucket; per-bucket policy permits writes scoped to the user's folder).
- "Sign out of all sessions" button calls
  `supabase.auth.signOut({ scope: 'global' })`.

## Acceptance criteria

- [ ] An unauthenticated visit to `/h/:id` redirects to `/auth/login`.
- [ ] An authenticated user with zero memberships is redirected to
      `/onboarding` from any `(app)` route.
- [ ] `app.profiles` row is auto-created by the trigger on signup.
- [ ] `redeem_invite` rejects expired and already-redeemed codes (client shows
      a typed error).
- [ ] An invite code is single-use; second redeem attempt fails.
- [ ] The Google OAuth button is rendered only when `VITE_FEATURE_GOOGLE_AUTH`
      is `'true'`.
- [ ] Household owners can transfer ownership; the last owner cannot leave
      without a transfer.
- [ ] Followers can read another household's recipes but UPDATE returns 0 rows
      (Playwright + RLS test).
- [ ] Profile changes persist and re-render the recipe view immediately
      (TanStack Query invalidates the recipes cache on profile change of
      `preferred_unit_system` or `preferred_language`).
- [ ] No emoji anywhere in this doc or governed code.
